- `form.resetFields()` 不会清除验证状态（需同时调用 `form.resetFields()` + `form.clearValidate()`）

## 🔶 中等问题

### M1. .catch(() => {}) 静默吞错误

多处代码使用 `.catch(() => {})` 完全吞掉表单验证错误，导致用户无法知道为什么操作没有生效：

| 文件 | 行号 | 代码 |
|------|------|------|
| `ClusterDeploy.tsx` | L715 | `form.validateFields().then(...).catch(() => {})` — 预览计划按钮 |
| `ClusterDeploy.tsx` | L718 | `form.validateFields().then(...).catch(() => {})` — 环境预检按钮 |

用户点击"预览计划"或"环境预检"时，如果表单验证不通过，没有任何提示。

### M2. 直接使用 api 对象绕过类型安全

以下页面使用 `import api from '../services/api'` 直接调用 API（而非通过 typed service 模块），失去了类型检查和响应拦截器的统一错误处理：

| `AlertManagePage.tsx` | `api.get('/alerts/templates')`, `api.post('/alerts/templates')` |

### M4. Dashboard 修改密码/重置密码 Modal 共用提交函数错误

**文件**: `frontend/src/pages/Dashboard.tsx` (L38-58)

`submitChangePassword` 函数在 `onOk` 中调用 `passwordForm.validateFields()`，但 Modal 的 `onOk` 直接绑定 `submitChangePassword`，如果表单验证失败，Modal 不会关闭但也没有错误提示。同时 `submitResetAllPasswords` 也有类似问题。

### M5. InstanceDetail 强改密码 Modal 表单验证问题

**文件**: `frontend/src/pages/InstanceDetail.tsx` (L366-387)

密码强度实时校验使用了 `forceResetForm.setFieldsValue` 设置虚拟字段 `_strength` 和 `_errors`，但这些字段不在 `Form.Item` 定义中，属于 hack 方式。`shouldUpdate` 渲染函数虽然能读取到，但方式不标准。

### M6. SecuritySettings 中 handleSave().catch() 模式混乱

**文件**: `frontend/src/pages/SecuritySettings.tsx` (L38)

```typescript
if (c) {
  handleSave().catch(() => {
    message.warning('请先填写密码');
```

`Switch` 的 `onChange` 中直接调用 `handleSave().catch()`，如果 `handleSave` 因为表单验证失败而 reject，用户看到的是"请先填写密码"提示但实际原因是表单验证失败，提示信息不准确。

## 🔹 轻微问题

### S1. 空表格本地化缺失

以下页面表格缺少 `locale` 的空数据配置，在无数据时显示默认英文 "No Data"：

| 文件 | 问题 |
|------|------|
| `BackupManage.tsx` | 策略表格无 `emptyText` |
| `AgentManage.tsx` | 执行结果表格无 `emptyText` |
| `InspectionPage.tsx` | 巡检报告表格无 `emptyText` |

### S2. 大量页面无 ErrorBoundary 包裹

所有页面组件均未使用 React ErrorBoundary 包裹，如果某个页面渲染出错，整个 Dashboard 布局会崩溃（白屏），没有降级UI。

### S3. 非必要 any 类型使用

大量组件使用 `any` 类型而非具体接口，导致 TypeScript 静态检查失效：

- `Dashboard.tsx`: `const [user, setUser] = useState<any>(null)` — 即使是宽松类型也应该定义 User 接口
- 多处 `res: any` 在 `catch` 块中捕获错误类型为 `any`，应使用 `unknown` 再类型断言

### S4. 使用 `key` 作为索引的潜在问题

多处表格的 `rowKey` 使用基于索引的 `key`，可能导致 React reconciliation 问题：

| 文件 | 行号 |
|------|------|
| `ClusterDeploy.tsx` | 计划预览 Modal 中的节点表格 `rowKey={(row: any) => row.id || row.host || Math.random()}` — 使用 `Math.random()` 会导致每次渲染都重新创建 DOM |
| `InstanceDetail.tsx` | 用户列表 `key: String(index)` |
| `DataStorage.tsx` | 迁移结果表格 `rowKey="table"` — 表名可能重复 |

### S5. 环境检查页面手动输入模式 Form.List layout 问题

**文件**: `frontend/src/pages/EnvironmentCheck.tsx` (L262-290)

手动输入模式使用 `Form.List` + `Space` 布局，但 `Space` 组件在 flex wrap 时换行会混乱，且 `MinusCircleOutlined` 点击删除后会导致后续 `Form.Item` 的 `name` 索引错乱。

### S6. MonitorDashboard 首次加载可能不显示指标

**文件**: `frontend/src/pages/MonitorDashboard.tsx` (L105-108)

`fetchMetrics` 只在 `selectedInstance` 变化时触发，但 `fetchInstances` 在首次加载时异步设置 `selectedInstance` 为第一个实例的 ID，这可能导致状态更新不同步，需要额外的 `useEffect`。

## 🛠️ 改进建议

### R1. 统一增加 ErrorBoundary

建议在每个页面组件外层包裹 ErrorBoundary，或至少给 Dashboard 添加全局 ErrorBoundary。

**推荐方案**:

```tsx
// 在 App.tsx 中添加
import { ErrorBoundary } from 'react-error-boundary'

<ErrorBoundary FallbackComponent={({error}) => <Result status="500" title="页面崩溃" subTitle={error.message} />}>
  <Suspense fallback={...}>
    <Routes>...</Routes>
  </Suspense>
</ErrorBoundary>
```

### R2. AlertManagePage 拆分 Form 实例

将 3 个 Modal 的 Form 拆分为独立实例，避免跨 Modal 状态污染：

```tsx
const [tplForm] = Form.useForm()
const [escForm] = Form.useForm()
const [silForm] = Form.useForm()
```

### R3. 统一错误处理策略

- 初始数据加载失败：应显示友好的空状态（Empty）+ 重试按钮，而非静默 `catch {}`
- 用户操作失败：应使用 `message.error` + 具体错误信息
- API 404：应在 interceptors 层面处理，或明确告知用户接口不可用

### R4. 为直接使用 `api` 的页面增加 typed service

将 `AIDiagnosisPage.tsx`、`AlertManagePage.tsx`、`DrillPage.tsx` 等页面迁移到使用 `services/api.ts` 中定义的 typed API 方法，获得类型安全和统一错误处理。

### R5. 修复 `.catch(() => {})` 问题

```tsx
// 改为
form.validateFields()
  .then(values => doPreview(arch, values))
  .catch(err => {
    // 表单验证失败时，antd 会自动显示错误提示，不需要额外处理
    // 但至少应该 console.warn 以便调试
    console.warn('Form validation failed:', err)
  })
```

### R6. 增加后端不可用时的降级处理

所有页面在 API 请求全部失败时，应显示清晰的提示，引导用户检查后端服务是否正常运行。目前大部分页面仅在控制台静默吞错，用户在 UI 上看到的是空白表格。

### R7. Login/AIDiagnosis/AlertManage 等页面考虑 SSR 兼容性

如果未来需要支持服务端渲染，直接使用 `api` 对象的页面需要适配。

## ⚠️ 运行时严重问题

### RB1. 系统设置页面 API 500 错误

**页面**: 系统设置 (`/dashboard/security-settings`)
**API**: `GET /api/v1/settings`
**状态码**: HTTP 500（出现 2 次）
**影响**: 系统设置页面加载时后端返回服务器错误，所有依赖 settings 的功能（查看密码保护、密码策略、监控阈值、平台参数、中继配置等）均无法正常初始化。
**建议**: 检查后端 `/api/v1/settings` 接口实现，确认该端点是否需要特定参数或数据库表支持。

### RB2. 后端 API ERR_ABORTED 请求失败（10+ 页面）

大量页面的 API 请求以 `net::ERR_ABORTED` 失败，说明后端可能未完全启动或某些接口不可用：

| 受影响页面 | 失败 API |
|-----------|----------|
| 总览、监控仪表盘、主机管理、集群部署、角色切换、拓扑视图 | `GET /api/v1/instances?limit=100&offset=0` |
| 环境检查（3次）、升级管理、数据迁移、审计日志、系统设置 | `GET /api/v1/instances?limit=1000&offset=0` |
| 高可用管理（2次） | `GET /api/v1/deployments?limit=1000&offset=0` |
| 审批管理 | `GET /api/v1/topology/clusters/mha-cluster-01/graph` |

**影响**: 页面虽然正常渲染，但数据表格为空。用户看到的是空白列表而不是数据。
**建议**: 
- 增加后端不可用时的降级提示（如 "无法连接到后端服务，请检查服务状态"）
- 对于 401/403 错误，应有明确的重新登录引导

### RB3. antd Modal `destroyOnClose` 废弃警告（20 个页面全部出现）

**错误信息**: `[antd: Modal] 'destroyOnClose' is deprecated. Please use 'destroyOnHidden' instead.`
**影响页面**: 所有包含 Modal 的页面（全部 20 个页面）
**受影响的组件文件**: `Dashboard.tsx`、`InstanceDetail.tsx`、`BackupManage.tsx`、`HostDetail.tsx` 等大量 Modal
**建议**: 全局搜索 `destroyOnClose` 替换为 `destroyOnHidden`。这是 antd v5 的 API 变更。

### RB4. antd Input `addonBefore` 废弃警告

**页面**: 系统设置
**错误信息**: `[antd: Input] 'addonBefore' is deprecated. Please use 'Space.Compact' instead.`
**受影响的组件文件**: `SecuritySettings.tsx` 中继配置部分的 Input 组件
**建议**: 将 `addonBefore` 属性替换为 `Space.Compact` 包裹方式。

### RB5. antd Spin `tip` 使用错误

**页面**: 监控仪表盘
**错误信息**: `[antd: Spin] 'tip' only work in nest or fullscreen pattern.`
**受影响的组件文件**: `MonitorDashboard.tsx` 中的 Spin 组件
**建议**: Spin 的 `tip` 属性仅在 `spinning` 包裹已渲染子元素时有效（nest 模式），或者设置 `fullscreen` 属性。建议将 Spin 包裹一个有内容的容器。

```tsx
  <App />
```

## ⚠️ 按钮点击触发的问题

### BC1. 系统设置 - 安装包管理"删除"按钮触发 API 500 和错误提示残留

**页面**: 系统设置 → 安装包管理标签
**按钮**: "**删除**"（安装包表格中的操作按钮）
**触发次数**: 3 次

**详细错误**:

| # | 错误类型 | 详情 |
|---|---------|------|
| 1 | `API 500` | `POST /api/v1/relay/delete-package` 返回 500 |
| 2 | `VISIBLE_ERROR` | "删除失败: delete failed: remove data\\packages\\mysql-8.4.0-linux-glibc2.28-x86_64.tar.xz: The system cannot find the file specified." |
| 3 | `VISIBLE_ERROR` | "删除失败: delete failed: remove data\\packages\\mysql-8.3.0-linux-glibc2.28-x86_64.tar.xz: The system cannot find the file specified." |
| 4 | `VISIBLE_ERROR` | "删除失败: delete failed: remove data\\packages\\mysql-5.7.44-linux-glibc2.12-x86_64.tar.gz: The system cannot find the file specified." |

**根本原因**:
- 安装包目录列表中的文件是预设的 MySQL 安装包 URL 列表，文件实际上不存在于服务器的 `data/packages/` 目录中
- 点击"删除"按钮尝试删除不存在的文件 → 后端返回 500 + 错误消息
- **更严重的问题**: 错误 toasts 在点击"下载"按钮后依然残留显示（"展开"按钮也会触发），说明错误提示没有被清理

**建议**:
1. 后端 `delete-package` 接口应对文件不存在的情况返回 404 而非 500
2. 前端应阻止对预设/不存在的文件调用删除接口
3. 错误 toast 应在页面切换或用户进行新操作时自动清除

### BC2. 系统设置 - antd message 静态函数上下文丢失

**页面**: 系统设置、备份管理
**按钮**: "**删除**"（系统设置安装包）、"**创建备份任务**"（备份管理）
**错误信息**:
```
Warning: [antd: message] Static function can not consume context like dynamic theme.
Please use 'App' component instead.
```

**原因**:
- 使用 `message.error()` 静态方法（在 `SecuritySettings.tsx` 和 `api.ts` 拦截器中）而不是从 `App.useApp()` 获取的 message 实例
- 静态 `message` 无法访问 antd 的动态主题上下文

**建议**:
- 在 `api.ts` 拦截器中改用从 `App.useApp()` 获取的 message 实例
- 或参考 `Login.tsx` 的做法：`const { message } = AntApp.useApp()`

### BC3. 数据存储 - Spin tip 使用错误

**页面**: 数据存储
**按钮**: "**执行迁移**"
**错误信息**:
```
Warning: [antd: Spin] 'tip' only work in nest or fullscreen pattern.
```

**原因**:
- `DataStorage.tsx` 中的 `<Spin tip="...">` 没有包裹子元素，或者子元素为 null/空
- `tip` 属性只有在 Spin 处于 nest 模式（有子元素）或 `fullscreen` 模式下才生效

**建议**:
- 确保 `<Spin>` 组件包裹了有效的子元素
- 或使用 `fullscreen` 属性（如果合适）

### BC4. 错误 toast 残留问题

**页面**: 系统设置
**问题**: 点击"删除"按钮后出现的错误 toast `"删除失败: ..."` 在后续点击"下载"和"展开"按钮时仍然显示在屏幕上

**现象**:
- 点击第一个"删除" → 出现错误 toast
- 点击"下载"（安装包）→ 错误 toast 仍然显示（没有替换为新的 toast）
- 点击"展开"（Yum/Apt 源）→ 旧的错误 toast 仍然残留

**建议**:
- 使用 `message.destroy()` 在发起新操作前先清除旧的 toast
- 或在 `Modal.confirm` 的 `onOk` 中不调用会触发 toast 的操作

## 🔶 注意事项

### BN1. 按钮不可点击原因分析

部分按钮未被点击（被脚本跳过），属于预期行为：

- **禁用按钮**: 如"批量检测"（需先勾选行）、"一键检测选中"（需先勾选行）、"执行选中主机"（需先选择主机） — 这些按钮后端不可用时应启用并显示提示而非禁用
- **隐藏按钮**: Popconfirm 包裹的按钮需要先触发 Popover，脚本未模拟确认流程
- **表单提交按钮**: 如"启动部署"、"执行切换"等需要填写表单后点击，脚本未填写表单

### BN2. 后端不可用导致按钮功能受限

后端 API 500 和 ERR_ABORTED 问题（详见第二部分 RB1/RB2）导致大多数数据操作类按钮（创建、删除、编辑）虽然在 UI 上可点击，但实际操作会失败——用户会看到空表格或 antd message 错误提示。
